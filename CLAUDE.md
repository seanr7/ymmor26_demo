# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Worktree & Branching Workflow (default)

Unless I say otherwise, follow this for every code-changing task. This is a solo repo with no CI/CD; pushes to the remote are just backups. All review and merging happens locally.

### 1. Start in a new worktree on a new branch

- Never edit directly on `main`. Check with `git branch --show-current` first.
- Create a worktree inside `.worktrees/` with a new branch:
```
git worktree add .worktrees/<short-desc> -b claude/<short-desc> main
```
- `cd` into it and work there. One task = one worktree = one branch.
- After creating the worktree, print its path so I can find it in the VS Code explorer under `.worktrees/`.

### 2. Work and self-review

- Commit in small, focused chunks with clear messages.
- Before declaring done, run the project's tests/linters. If unclear what those are, ask.
- Then summarize for me: branch name, worktree path, `git log main..HEAD --oneline`, and `git diff main --stat`. Wait for my approval before merging.

### 3. Merge locally, then push for backup

- From the main repo directory (not the worktree):
```
git checkout main
git merge --no-ff claude/<short-desc>
git push origin main
```
- Then clean up:
```
git worktree remove .worktrees/<short-desc>
git branch -d claude/<short-desc>
```
- No PRs, no tags, no other remote actions.

### 4. Parallel tasks

If I start a second task while another worktree is still open, create a fresh worktree for it rather than reusing one.

### 5. Escape hatches

- If I say "stay on this branch", "work in place", or "quick fix", skip the worktree step.
- If the workflow is blocked (dirty `main`, worktree path exists, branch name taken), stop and tell me before improvising.

## This Project

Measure how the data efficiency of learning an ODE vector field with an MLP depends jointly on the spatial dimension D and the number of training trajectories N. The deliverable is rollout error as a function of these two axes.

## System

Ground truth is the Lorenz-96 system,

$$\frac{dx_i}{dt} = (x_{i+1} - x_{i-2})\,x_{i-1} - x_i + F, \qquad i = 1,\dots,D,$$

with periodic indices and F = 8 (chaotic regime). Lorenz-96 is chosen because D is a free parameter while the form of the dynamics is unchanged, the attractor has approximately D-independent marginal statistics, and the Lyapunov time is approximately D-independent — so rollout errors and horizons are comparable across D.

## Model

A small MLP f_theta : R^D -> R^D trained as a one-step residual integrator,

$$\hat{x}_{t+\Delta t} = x_t + \Delta t \cdot f_\theta(x_t).$$

Architecture is held fixed across all D. Inputs and target residuals are standardized using attractor statistics.

## Experimental Axes

- **Spatial dimension** D: a small set of values spanning a meaningful range.
- **Number of training trajectories** N: a sweep from very few to moderately many.

Training trajectories are generated by sampling initial conditions from a long spin-up trajectory (from the invariant measure) and integrating each forward for a fixed horizon T_train with a high-accuracy reference integrator. T_train is fixed so that N genuinely controls initial-condition coverage of the attractor. N counts trajectories, not one-step pairs.

## Evaluation

Held-out test ICs rolled out autoregressively (no teacher forcing). Metric is relative L2 error averaged over test ICs and the full rollout horizon (several Lyapunov times), giving one scalar per (D, N) cell. Final deliverable is a heatmap over the (D, N) grid.

## Stack

- JAX, Flax, Optax for modeling and training
- Hydra with structured configs (ConfigStore) for configuration
- Matplotlib for plotting (use repo plotting helpers when available)

## Code Style

- **Functions over classes.** No class hierarchies. Exceptions: `flax.nn.Module` and `@dataclass` configs for Hydra.
- **No defensive coding.** No input validation, no try/except. Shape-check `assert`s are fine.
- **Type hints optional.** Use only when they clarify something non-obvious. Never use Protocol, Generic, TypeVar, or complex unions.
- **Logging is `print()`.** Lowercase, one line, progress-oriented.
- **Functional JAX:** explicit inputs (params, batch, rng_key), explicit outputs.
- **Don't reformat** unrelated code. Don't add dependencies unless asked.
- **Standard names:** `logits`, `params`, `grads`, `state`, `batch`, `rng_key`, `apply_fn`, `loss`.

## Hydra Conventions

- Each logical group (network, optimizer, data, loss, ...) is a separate `@dataclass` with plain-field defaults.
- Top-level `Config` composes groups with `field(default_factory=GroupName)`.
- Per-dataset configs are built by instantiating `Config(...)` with overrides and registered via `cs.store(name=..., node=...)`.
- Entrypoint: `@hydra.main(version_base=None, config_name="default")` taking `cfg: Config`.
- Output paths come from `get_outpath()`. Don't reconstruct paths from `cfg.name`.

## Plotting

- Use existing plotting helpers (see `.claude/skills/plotting/SKILL.md` for the full table) before writing raw matplotlib.
- If data has a time dimension, default to animation (gif), not a static frame.
- Save to `get_outpath()` and print the path. Never save to cwd.
- No `'jet'` colormap. Use `'viridis'` for scalar fields, `'RdBu_r'` for signed data centered at zero.
- Always set explicit `figsize`, axis labels, and `tight_layout()`.
